# Product Requirements Document (PRD)

В этом документе зафиксированы функциональные и нефункциональные требования к сервису «Red Pen» (redpen-checker). Они опираются на анализ персон и должны использоваться для планирования спринтов, создания задач и приёмки.

## Функциональные требования (MVP)

### US-01: Быстрая авторизация
**Как** учитель,  
**я хочу** авторизоваться через свой Google-аккаунт или по электронной почте школы,  
**чтобы** мой список классов был защищён без запоминания нового пароля.

Критерии приёмки:
- Поддержка входа через Google OAuth или Email+OTP.
- После первого входа создаётся учётная запись Teacher, выпускается API-ключ.
- API-ключ можно использовать для доступа к методам сервиса.

### US-02: Импорт класса со списком учеников
**Как** учитель,  
**я хочу** импортировать класс из CSV-файла или вставить список учеников через текстовое поле,  
**чтобы** не вводить 35+ фамилий вручную.

Критерии приёмки:
- Возможность создать класс с именем (например, «2А»).
- Загрузка CSV с колонками «ФИО» или вставка списка (одна строка — один ученик).
- Автоматическое определение порядка учеников.
- Сохранение класса и учеников в БД.

### US-03: Загрузка и проверка одной работы с возможностью корректировки
**Как** учитель,  
**я хочу** загрузить фото тетради и получить проверенный текст с подчёркнутыми ошибками и краткими правилами,  
**чтобы** при необходимости скорректировать распознавание, выставить оценку и сохранить результат.

Критерии приёмки:
- Загрузка изображения через web-интерфейс (JPG/PNG до 10 МБ).
- Отправка в GigaChat OCR API, получение распознанного текста.
- Отправка текста в DeepSeek API с промптом, учитывающим предмет и задание.
- Отображение результата: оригинальный текст, список ошибок с пояснениями.
- Возможность вручную исправить распознанный текст или снять/добавить ошибку.
- Кнопка «Выставить оценку» с автоматическим предложением на основе количества ошибок.
- Сохранение результата в БД.

### US-04: Последовательная пакетная проверка
**Как** учитель,  
**я хочу** загрузить несколько работ учеников (до 10 за раз) и поочерёдно просматривать результаты их проверки в едином интерфейсе,  
**чтобы** ускорить процесс и не переключаться между страницами.

Критерии приёмки:
- Создание сессии проверки с указанием класса, предмета, задания и эталона.
- Последовательная загрузка фото для каждого ученика (с подсветкой текущего ученика).
- Возможность пропустить ученика или переснять фото.
- После загрузки — асинхронная обработка всех работ.
- Просмотр результатов в виде таблицы с фильтрацией и возможностью перехода к деталям.

### US-05: История проверок
**Как** учитель,  
**я хочу** видеть историю проверок по каждому ученику,  
**чтобы** отслеживать прогресс и при необходимости показать работу родителю.

Критерии приёмки:
- Фильтр по классу, ученику, предмету, дате.
- Просмотр сохранённых результатов (текст, ошибки, оценка).
- Возможность скачать PDF-отчёт для родительского собрания.

## Нефункциональные требования

### NR-01: Время отклика
Время от отправки фото до появления проверенного текста не должно превышать **10 секунд** для 90% запросов (P95).

### NR-02: Адаптивность интерфейса
Интерфейс должен быть полностью функционален на экранах от 320px (смартфон) до 1440px (ноутбук). Приоритет — mobile-first, так как учитель активно использует планшет.

### NR-03: Точность распознавания ошибок
Точность (Precision/Recall) обнаружения орфографических ошибок должна быть не ниже **95%** на тестовом наборе реальных школьных работ.

### NR-04: Офлайн-загрузка
Должна быть возможность инициировать загрузку фото без активного интернет-соединения с последующей автоматической синхронизацией при появлении сети.

### NR-05: Безопасность и соответствие законодательству
Данные учеников и фото должны храниться с соблюдением требований 152-ФЗ:
- Шифрование данных при передаче (TLS 1.3) и хранении.
- Обезличивание по требованию.
- Все секреты и ключи хранятся в защищённом хранилище (переменные окружения, секреты GitHub Actions), никогда — в коде.

### NR-06: Масштабируемость
Система должна автоматически масштабироваться для обработки пиковых нагрузок в вечернее время (ориентировочно 19:00–22:00 по МСК) и обеспечивать доступность не ниже 99,5%.
